# ROOT/domain/codefromspec/specifications

## Intent

Defines the structure of specifications as relevant to this tool:
nodes, logical names, frontmatter format, and leaf and test node
contracts.

## Contracts

### Location

Specifications live under `code-from-spec/spec/`.

### Nodes

Every spec node is a directory containing a `_node.md` file. The
directory structure is the spec tree — a node's position in the
filesystem is its position in the hierarchy.

A node with child directories is an **intermediate node**. A node
without children is a **leaf node**. Only leaf nodes may have
`implements`.

A **test node** is a file ending in `.test.md` placed inside a leaf
node's directory. The canonical test node is `default.test.md`.
Additional test nodes use `<name>.test.md`. Only leaf nodes may have
test nodes — test nodes alongside intermediate nodes are invalid.

```
config/
  _node.md             ← spec node (leaf)
  default.test.md      ← canonical test node
  edge_cases.test.md   ← additional test node
```

### Logical names

Every node has a logical name derived from its position in the tree.
Spec nodes use the `ROOT/` prefix; test nodes use the `TEST/` prefix.

| Logical name | Resolves to |
|---|---|
| `ROOT` | `code-from-spec/spec/_node.md` |
| `ROOT/x/y` | `code-from-spec/spec/x/y/_node.md` |
| `TEST/x/y` | `code-from-spec/spec/x/y/default.test.md` |
| `TEST/x/y(name)` | `code-from-spec/spec/x/y/name.test.md` |

Resolution rules:
- `ROOT` → `code-from-spec/spec/_node.md`
- `ROOT/<path>` → `code-from-spec/spec/<path>/_node.md`
- `TEST/<path>` → `code-from-spec/spec/<path>/default.test.md`
- `TEST/<path>(<name>)` → `code-from-spec/spec/<path>/<name>.test.md`

### Parent relationships

- The root node (`ROOT`) has no parent.
- A spec node's parent is determined by its logical name:
  `ROOT/x/y/z` has parent `ROOT/x/y`.
- A test node's parent is the `_node.md` in the same directory:
  `TEST/x/y` and `TEST/x/y(name)` both have parent `ROOT/x/y`.

### Frontmatter

Every node begins with a YAML frontmatter block. The following
fields are relevant to this tool:

| Field | Description | Notes |
|---|---|---|
| `version` | Current version of this node. | All nodes |
| `depends_on` | Cross-tree dependencies with their known versions. | Optional |
| `implements` | Source files generated by this node. | Leaf and test nodes only |

Other frontmatter fields may exist and must be ignored.

### depends_on structure

Each `depends_on` entry has:

| Field | Type | Required | Description |
|---|---|---|---|
| `path` | string | yes | Logical name of the dependency (`ROOT/` or `EXTERNAL/`). |
| `version` | int | yes | Expected version of the dependency. |
| `filter` | []string | no | Glob patterns selecting files from an external dependency folder, in addition to its `_external.md`. Only meaningful for `EXTERNAL/` paths. |

Example — leaf node with mixed dependencies:

```yaml
---
version: 3
depends_on:
  - path: ROOT/architecture/config
    version: 2
  - path: EXTERNAL/celcoin-api
    version: 5
    filter:
      - "endpoints/*.md"
      - "types.md"
implements:
  - internal/payments/processor.go
---
```

Example — test node:

```yaml
---
version: 1
parent_version: 2
implements:
  - internal/payments/processor_test.go
---
```

### implements

The `implements` field lists the source files this node generates,
as paths relative to the project root. This list is the authoritative
scope of what this node may produce — no file outside it should be
written during generation.
